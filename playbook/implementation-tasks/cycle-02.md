# cycle-02 implementation tasks

## ゴール
複数画像アップロードをキュー制御付きで安定化し、投稿一覧の自動更新とスライドショーの最小導線を動作させる。

## 前提
- 基本設計は architecture.md を唯一の設計基準とする
- cycle-01 の direct upload と post 状態遷移が動作済みである
- 今回は画像のみを対象とし、動画は扱わない
- 既存構成を壊さない
- 不要な依存を追加しない

## 実装タスク

### 1. posts API の差分取得を仕上げる
- GET /api/rooms/:roomId/posts の since 対応を安定化する
- uploaded かつ visible のみを返す条件を再確認する
- limit の上限値を決めて過剰取得を防ぐ
- serverTime を返し、次回 since に使えるようにする
- created_at 昇順を基本順序として固定する

完了条件
- 初回全件取得と差分取得の両方が動く
- pending / failed / hidden が混ざらない
- フロントが serverTime を保存して次回取得に使える

### 2. viewUrl 一括発行 API を実装する
- POST /api/rooms/:roomId/posts/view-urls を追加する
- request body で postIds を受け取る
- 対象 post が roomId に属するか確認する
- uploaded かつ visible の post のみに限定して短命 GET URL を発行する
- 返却形式を postId => viewUrl の対応が分かる形にする
- 不正 postId や期限切れ room のエラーを返す

完了条件
- 複数 postId に対して一括で短命 URL を返せる
- 非表示や未完了投稿のURLは発行されない

### 3. スライドショー設定取得 API を接続する
- GET /api/rooms/:roomId/slideshow-settings を cycle-02 の利用レベルで確認する
- room 作成時の初期値が必ず存在することを再確認する
- interval_seconds と show_nickname をフロントで使える形に整える
- order_mode は asc を前提に扱い、manual 専用処理はまだ実装しない

完了条件
- スライドショー画面が設定値を取得して利用できる

### 4. フロントのアップロードキューを実装する
- 参加者画面のファイル選択を複数対応にする
- 画像のみ選択可能に制限する
- QueueItem の状態モデルを作る
  - pending
  - uploading
  - completing
  - done
  - error
- 同時アップロード数3件の実行制御を実装する
- 1件ごとの処理は cycle-01 の upload-url → R2 PUT → complete / fail を再利用する
- 失敗したファイルだけ個別リトライできるようにする
- 成功した投稿は完了時に一覧へ即時反映する

完了条件
- 複数画像選択で順次または並列に処理される
- 失敗ファイルのみ再試行できる
- ほかの処理中ファイルに巻き込まれず継続できる

### 5. 参加者画面の投稿一覧ポーリングを実装する
- 初回ロード時に投稿一覧を取得する
- 3〜5秒間隔のポーリングを実装する
- since と serverTime を使って差分取得する
- 既存一覧へのマージ処理を実装する
- 同一 postId の重複追加を防ぐ
- unmount 時に interval を確実に解除する

完了条件
- 別端末または別操作で増えた投稿が自動で見える
- 一覧が重複しない

### 6. スライドショー画面を実装する
- ルート /room/:roomId/slideshow を追加する
- room 情報と slideshow-settings を取得する
- 初回投稿一覧取得と定期ポーリングを実装する
- 取得した post から表示対象配列を作る
- 投稿0件時の空状態を表示する
- interval_seconds ごとに現在表示 index を進める
- show_nickname が true のときだけニックネーム表示を出す

完了条件
- スライドショー画面で画像が順番に切り替わる
- 投稿0件でも破綻しない

### 7. スライドショー向けの viewUrl キャッシュと再取得を実装する
- 表示対象 post に必要な viewUrl をまとめて取得する
- postId と URL と期限の管理構造を作る
- URL がない、または期限切れが近い場合に再取得する
- 一括取得件数は最小限に絞る
- URL取得失敗時はスライドショーを止めず、次回ポーリングまたは再試行で回復できるようにする

完了条件
- 画像切替のたびに個別 API を叩かなくても表示できる
- 一定時間表示しても URL 期限切れで全面エラーになりにくい

### 8. UI の最低限改善を入れる
- アップロードキュー一覧に状態表示を出す
- 処理中件数 / 完了件数 / 失敗件数の簡易表示を出す
- スライドショー遷移ボタンを参加者画面に置く
- スライドショーの画像切替に最小限のフェードを入れてよい

完了条件
- ユーザーが今何が起きているか分かる
- スライドショーへ迷わず遷移できる

### 9. テストと確認を追加する
- API 側確認
  - posts difference fetch
  - view-urls validation
  - slideshow-settings fetch
- フロント側確認
  - 複数画像キュー処理
  - 一部失敗時の継続
  - ポーリング更新
  - スライドショー表示
- build / lint / typecheck を通す
- 手動確認を行う
  - 画像3〜5枚の連続アップロード
  - 失敗ファイルのリトライ
  - 投稿0件状態のスライドショー
  - 追加投稿後の自動反映

完了条件
- lint / typecheck / build が通る
- 対象ユースケースが縦に動く

## 実装順の推奨
1. posts 差分取得と view-urls API
2. 参加者画面の複数画像キュー
3. 投稿一覧ポーリング
4. スライドショー画面
5. viewUrl キャッシュと再取得
6. テスト・仕上げ

## 注意事項
- API サーバー経由でファイル本体を中継しない
- 動画対応に広げない
- スライドショー用URLを毎回1件ずつ個別取得しない
- 一覧マージ時に postId 重複を防ぐ
- cycle-01 の upload 状態遷移を壊さない
- order_mode manual のUIや並び替え編集には手を出さない

## このサイクルの完了判定
- 複数画像アップロードが3件並列まで安定して動く
- 一部失敗時も全体処理が止まらない
- 投稿一覧が差分ポーリングで自動更新される
- スライドショー画面で画像が一定間隔で順次表示される
- viewUrl 一括発行を利用し、表示が過度にカクつかない
- build が通る

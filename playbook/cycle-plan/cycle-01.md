project: wedding-photo-share
cycle: 01
goal: ルーム作成から参加者入室、画像1枚の直アップロード完了と投稿一覧反映までを縦に動かす

## Scope（Must）
1. ルーム作成の最小導線
- 管理者がトップ画面からルームを作成できる
- ルーム作成APIが rooms / slideshow_settings の初期データを保存できる
- ルーム作成完了後に participantUrl / roomId / hostToken を返せる
- 管理者画面で participantUrl を確認できる

2. 参加者入室の最小導線
- 参加者が /room/:roomId にアクセスできる
- ルーム情報取得APIで、存在確認・有効期限確認ができる
- passcode 設定時は入力と検証ができる
- ニックネーム入力後に localStorage 保存し、再訪時に再入力を省略できる

3. 画像1枚アップロードの最小縦切り
- 参加者が画像1枚を選択できる
- upload-url API が MIME / size / room 有効期限を検証できる
- upload-url 発行時に posts の仮レコードを pending で作成できる
- フロントが署名付きURLで R2 へ直接 PUT できる
- complete API で upload_status を uploaded に更新できる
- fail API で upload_status を failed に更新できる
- 投稿一覧APIで uploaded かつ visible の画像だけ取得できる
- アップロード完了後、参加者画面の一覧に即時反映できる

4. 最小UI/状態表示
- アップロード中 / 完了 / 失敗 の状態を表示する
- 失敗時に手動リトライできる
- ルーム期限切れ時のエラー画面を出せる

## Scope（Should）
- 参加者画面で自分の投稿であることが分かる簡易表示
- 管理者画面に QR 表示用のプレースホルダを置く
- 開発用 seed または簡易デバッグ導線を用意する

## Out of Scope（このサイクルではやらない）
- 動画アップロード
- 複数ファイル同時アップロード
- 画像3件キュー / 動画1件キューの本実装
- スライドショー画面
- 投稿の非表示 / 削除
- 表示順制御
- 一括ダウンロード
- Cron による TTL 削除本実装
- viewUrl 一括発行 API の本実装
- 本番品質のデザイン磨き込み

## 技術方針
- API は Cloudflare Workers + Hono 前提で実装する
- DB は D1、ストレージは R2 を使う
- フロントは Pages 想定の React / Vite 構成で進める
- まずは画像のみ、1件アップロード成功を最優先にする
- 認証は host_token / passcode の最小検証に留める

## 完了条件
- 管理者がルームを作成できる
- 参加者が room URL から入室し、ニックネーム入力後に画像1枚をアップロードできる
- 画像ファイルが API サーバーを経由せず R2 に保存される
- posts の仮登録 → 完了確定 → 一覧反映の整合性が崩れない
- 失敗時に failed 状態へ遷移し、再試行できる
- build が通る
- 既存構成を壊していない

## 人手確認
- hostToken を UI 上でどう見せるかは暫定でよいが、紛失時再発行しない方針を維持できているか
- passcode あり / なし両方で入室導線が破綻していないか
- スマホ実機で画像1枚アップロードが極端に詰まらないか

# 昇格基準 (Promotion Criteria)

B -> C への昇格は **必要になるまで行わない**。このドキュメントでは、昇格が必要なケースと不要なケースを具体例とともに示す。

---

## 昇格が必要なケース

### 1. データ共有が必要

**状況:** 複数端末、または複数ユーザーで同じデータを参照・編集したい

**具体例:**
- チームでタスクを共有したい
- スマホで入力したデータをPCで見たい
- 家族で買い物リストを共有したい

**判定:**
```
Q: 同じユーザーの複数端末か？
  → Yes: 同期機能（CRDTやP2P）で対応できないか検討
  → No（複数ユーザー）: 昇格が必要
```

### 2. サーバー処理が必要

**状況:** クライアント側では実行できない、または実行すべきでない処理がある

**具体例:**
- 課金処理（Stripe Webhook受け取り）
- 外部APIの秘密鍵を使った呼び出し
- 重いバッチ処理（大量データの集計）
- メール送信、プッシュ通知

**判定:**
```
Q: 秘密情報（APIキー等）が必要か？
  → Yes: 昇格が必要
Q: 処理時間がユーザー体感に影響するか？
  → Yes: バックグラウンド処理として昇格を検討
```

### 3. 認証・権限が必要

**状況:** ユーザーを識別し、権限に応じてアクセスを制御したい

**具体例:**
- ログインして自分のデータだけ見たい
- 管理者だけが設定を変更できるようにしたい
- 有料ユーザーだけが使える機能がある

**判定:**
```
Q: ユーザーIDをサーバーで検証する必要があるか？
  → Yes: 昇格が必要
Q: ローカルで「見せない」だけでいいか？
  → Yes: B側でUIを出し分けるだけで対応可能
```

### 4. データ永続性が重要

**状況:** ブラウザのストレージでは消失リスクが許容できない

**具体例:**
- 長期間の記録（健康データ、家計簿）
- 消えたら困るドキュメント
- 法的な保存義務があるデータ

**判定:**
```
Q: エクスポート/インポートで対応できるか？
  → Yes: B側で対応
  → No: 昇格が必要
```

---

## 昇格しないケース

以下のケースでは、まずB側で対応することを検討する。

| 要望 | B側での対応 |
|------|-------------|
| 「いつかマルチユーザーにするかも」 | 不要。必要になってから |
| 「バックアップが欲しい」 | エクスポート/インポート機能 |
| 「PWAが遅い」 | パフォーマンス最適化（仮想スクロール、遅延読み込み等） |
| 「オフラインでも使いたい」 | PWAのService Worker（Bの領域） |
| 「データ量が多い」 | IndexedDBは十分大きい（数GB） |
| 「同じ端末で複数アカウント」 | ローカルで切り替え機能 |

## 判定フローチャート

```
[スタート]
    ↓
複数ユーザーでデータ共有が必要？
    ├─ Yes → 昇格
    ↓ No
秘密情報を使った処理が必要？
    ├─ Yes → 昇格
    ↓ No
サーバー検証が必要な認証/権限？
    ├─ Yes → 昇格
    ↓ No
データ消失が致命的で、エクスポートでは不十分？
    ├─ Yes → 昇格
    ↓ No
[Bのまま]
```

---

## 昇格決定時のチェック

昇格を決めたら、以下を確認:

1. **B側の準備**
   - [ ] Repository interface が実装されている
   - [ ] Auth injection point が準備されている
   - [ ] データのエクスポート機能がある（移行用）

2. **記録**
   - [ ] `docs/decision_log.md` に昇格理由を記載
   - [ ] 選定した技術スタックを記載

3. **リスク確認**
   - [ ] 運用コストを見積もった
   - [ ] セキュリティ責任を理解した
